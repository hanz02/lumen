EVIDENCE 2 — The actual math, src/sun/solar.ts lines 99-131 (RAW). These are the
NOAA Solar Calculator / Meeus "Astronomical Algorithms" low-order coefficients,
NOT the NREL SPA's VSOP87 periodic-term tables:

  // Geometric mean longitude / anomaly of the sun, orbit eccentricity
  const L0 = norm360(280.46646 + T * (36000.76983 + T * 0.0003032));
  const M = 357.52911 + T * (35999.05029 - 0.0001537 * T);
  const e = 0.016708634 - T * (0.000042037 + 0.0000001267 * T);

  // Equation of center -> true & apparent longitude (nutation/aberration)
  const C =
    Math.sin(toRad(M)) * (1.914602 - T * (0.004817 + 0.000014 * T)) +
    Math.sin(toRad(2 * M)) * (0.019993 - 0.000101 * T) +
    Math.sin(toRad(3 * M)) * 0.000289;
  const omega = 125.04 - 1934.136 * T;
  const lambda = L0 + C - 0.00569 - 0.00478 * Math.sin(toRad(omega));
  ...
  // Equation of time (minutes)
  const y = Math.tan(toRad(eps / 2)) ** 2;

WHY THIS IS NOT REDA & ANDREAS:
- The NREL SPA (Reda & Andreas 2004) computes heliocentric longitude/latitude/
  radius from the VSOP87 tables (L0..L5, B0..B1, R0..R4 — hundreds of periodic
  terms) and a 63-term nutation series, reaching ±0.0003°.
- The code above uses a SINGLE mean-longitude term (280.46646...), a 3-term
  equation-of-centre, and the y=tan^2(eps/2) equation-of-time. These are verbatim
  the NOAA Solar Position Calculator equations (derived from Meeus, "Astronomical
  Algorithms"), accuracy ~±0.01°. No VSOP87 tables are present anywhere in the repo.

EVIDENCE 5 — RATIONALE for choosing NOAA/Meeus over the full Reda & Andreas (NREL)
SPA. INSTRUCTIONS FOR CLAUDE CHAT: write Chapter 3's justification using ONLY the
points below. Do not invent benchmark numbers, runtimes, file sizes, or citations
that are not stated here. Two of the four points are DIRECTLY SOURCED from the
repository (marked SOURCED); two are ENGINEERING INFERENCE from the same source
material, not verbatim repo text (marked INFERENCE) — phrase those as reasoning
the author applied, not as a quoted fact from the code.
--------------------------------------------------------------------------------

(a) [SOURCED — verbatim from src/sun/solar.ts header comment, already quoted in
    EVIDENCE 1 above] Accuracy-matching argument: NOAA/Meeus gives ~±0.01°
    accuracy (valid 1900-2100). This is ~3 orders of magnitude finer than the
    two real error sources downstream of it in this app's pipeline:
      - the window-facing model uses 90-degree aspect sectors (north/east/south/
        west facing), and
      - the phone compass carries ~5-10 degrees of error.
    Therefore the full NREL SPA's ±0.0003° accuracy would not be "spent" anywhere
    — no downstream computation in this app can use precision finer than what
    the compass and aspect-sector model already throw away. This is the ONLY
    justification the codebase itself states in writing.

(b) [INFERENCE — reasoning the author can state about implementation cost, not
    quoted from any code comment] Implementation-complexity argument: the full
    Reda & Andreas (2004) NREL SPA requires the VSOP87 periodic-term tables for
    heliocentric longitude, latitude and radius (on the order of hundreds of
    coefficients across multiple series) plus a multi-term nutation series, to
    reach its ±0.0003° accuracy. The NOAA/Meeus formulation needs only a small
    number of polynomial terms (the geometric mean longitude, mean anomaly,
    orbital eccentricity, a 3-term equation-of-centre, and the equation-of-time
    expression — all visible in EVIDENCE 2 above). For a single-developer FYP
    module that has to be hand-coded in TypeScript, unit-tested, and verified
    against known ephemeris values without an external astronomy library, the
    smaller coefficient set is materially lower-risk to implement correctly and
    easier to verify line-by-line than transcribing and validating the VSOP87
    tables. DO NOT state a specific term count, line count, or error-rate
    estimate for this point — only the qualitative claim that it is meaningfully
    smaller and simpler to implement and verify by hand.

(c) [INFERENCE] No external dependency: solar.ts is documented as "pure
    functions, no react-native imports" (see EVIDENCE 1, last paragraph). A
    hand-rolled small-coefficient formulation keeps the whole SPA self-contained
    in one TypeScript file; a full VSOP87 implementation would still be possible
    in pure TypeScript but would substantially enlarge that one module for a
    precision gain point (a) shows is not usable downstream.

(d) [INFERENCE] This is a precision/complexity trade-off, not a correctness
    shortcut: Meeus's "Astronomical Algorithms" and the NOAA Solar Calculator
    are themselves standard reference formulations for solar position (the
    code comment in EVIDENCE 1 attributes the implementation to them by name).
    Reda & Andreas (2004) is retained in the thesis as the upper-bound reference
    SPA specifically to justify, by comparison, why ±0.01° was judged sufficient
    here — i.e., R&A is cited to BOUND the accuracy claim, not because it was
    implemented.
