================================================================================
DOUBT D2 — What does the app do on E_AR_UNSUPPORTED? Dialog or silent grey-out?
================================================================================
ANSWER: BOTH, at different moments. There are TWO distinct UX paths, not one.

PATH 1 — PROACTIVE (checked at startup, before the user ever taps an AR button):
  App.tsx:258-265
    checkARAvailability().then((status) => {
      if (status === 'UNSUPPORTED_DEVICE_NOT_CAPABLE') setArUnsupported(true);
    });
  Setting arUnsupported=true greys out the AR buttons (arUnsupported is passed as
  a prop into the capture cards, App.tsx:911, 924) AND triggers a ONE-TIME MODAL
  via a separate effect (App.tsx:215-220):
    useEffect(() => {
      if (arUnsupported && step === SPOT && !arUnsupportedModalShownRef.current) {
        arUnsupportedModalShownRef.current = true;
        setShowArUnsupportedModal(true);
      }
    }, [arUnsupported, step]);
  The modal itself (App.tsx:1122-1131) is a real ConfirmModal, not a toast:
    <ConfirmModal
      visible={showArUnsupportedModal}
      icon="alert"
      hideCancel
      title="AR not available on this device"
      message="This device doesn't support ARCore, so the AR measurement buttons
        are disabled. Use the tape-measure fields to enter the plant-to-window
        distance and window dimensions manually — everything else in the app
        works normally."
      confirmLabel="Got it, I'll use tape"
      onConfirm={() => setShowArUnsupportedModal(false)}
      onCancel={() => setShowArUnsupportedModal(false)}
    />

PATH 2 — REACTIVE (mid-session failure, e.g. a device that reports
"SUPPORTED_NOT_INSTALLED" at checkAvailability time but actually fails when the
AR activity itself tries to start — the field-confirmed Redmi Note 10 case):
  App.tsx:679-686 (measureDistance catch block) and :723-730 (measureWindowDim
  catch block), both identical pattern:
    if (error?.code === 'E_AR_UNSUPPORTED') {
      setArUnsupported(true);
      showToast({
        title: 'AR not supported on this device',
        message: 'Use the tape-measurement fields instead.',
        variant: 'info',
      });
    }
  This shows a TOAST immediately. Because setArUnsupported(true) also flips the
  same arUnsupported state used by Path 1's effect, if the user is on the SPOT
  step and the modal has not already been shown this session, the one-time modal
  ALSO appears right after (the effect re-fires on the arUnsupported state
  change). If the modal was already shown earlier in the session, only the toast
  appears for subsequent failures.

VERDICT: The app does NOT silently grey out the button with no explanation. It
(a) proactively greys out AR buttons and shows an explanatory ConfirmModal at
startup if the device is confirmed incapable, and (b) shows an immediate Toast
(plus the same modal, once per session, if not already seen) when an AR call
fails live with E_AR_UNSUPPORTED. State BOTH paths in Chapter 4 — do not collapse
them into one behaviour.
